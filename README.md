# LunchBot

Телеграм-бот для заказа обедов: опрос по меню, проверка чеков об оплате, каталог блюд с фото.

## Что нового по сравнению с исходной версией

- **Проверка чеков.** Перед голосованием человек шлёт боту в личку чек (PDF или фото).
  Бот извлекает текст, проверяет получателя и дату, и только потом открывает голосование.
- **Каталог блюд.** Постоянные блюда хранятся в `dish` с фото. Блюда дня в каталог не попадают.
- **`/lunch` работает всегда**, независимо от состояния каталога.

## Быстрый старт

### 1. Настройки

```bash
cp .env.example .env
```

Заполните `.env`:

| Переменная | Где взять |
|---|---|
| `TELEGRAM_BOT_TOKEN` | @BotFather |
| `LUNCH_ADMIN_CHAT_ID` | ваш `user_id` — напишите @userinfobot |

Username бота задавать не нужно: бот узнаёт его сам через `getMe`.

Реквизит получателя в `.env` **не задаётся** — он живёт в БД и меняется командой
`/recipient` без рестарта.

`LUNCH_ADMIN_CHAT_ID` нужен только для **чеков** — туда приходит очередь
на ручную проверку. Каталог правят администраторы группы, а не владелец этого id.

### 2. Запуск в Docker

```bash
docker compose up --build
```

Поднимутся Postgres и бот. Flyway накатит миграции V1–V4 сам.

### 3. Запуск локально (без Docker)

Нужен Postgres на `localhost:5432` и `tesseract-ocr` с русским языком:

```bash
# Ubuntu / Debian
sudo apt install tesseract-ocr tesseract-ocr-rus

# macOS
brew install tesseract tesseract-lang
export LUNCH_OCR_DATAPATH=/opt/homebrew/share/tessdata
```

Затем:

```bash
export $(cat .env | xargs)
./mvnw spring-boot:run
```

Без tesseract бот **не упадёт**: PDF-чеки будут проверяться, фото уйдут админу на ручное подтверждение.

## Команды

| Команда | Где | Кто |
|---|---|---|
| `/lunch` + список блюд | группа | админ чата |
| `/close` | группа | админ |
| `/summary` | группа | админ |
| `/menu` | группа | все |
| `/catalog` | группа | админ чата |
| `/recipient <тел> <Имя Фамилия>` | группа | админ чата |
| `/dish <название>` + фото | группа | админ чата |
| `/cancel` | группа | выход из диалога каталога |

Чеки принимаются **только в личке** и **только в формате PDF** — в группе их
видели бы все, а PDF не грузит ноутбук (в отличие от распознавания фото).
Всё остальное живёт в группе.

Формат `/lunch` — как присылают в чате, каждое блюдо с новой строки:

```
/lunch
Здравствуйте 🌹🌹🌹

Босо лагман
Бифштекс гарнир
Том ям 🍠 с курицей 🍚 рис
```

Строка приветствия и пустые строки отбрасываются автоматически.

## Как работает проверка чека

```
Кнопка «🧾 Подтвердить оплату»  ->  t.me/BOT?start=pay_<pollId>
        ->  человек шлёт PDF или фото в личку
        ->  PDFBox (текстовый слой) либо tesseract (OCR)
        ->  ReceiptVerifier: телефон получателя + дата
```

Три исхода:

| | |
|---|---|
| `APPROVED` | телефон получателя **и** дата в допустимом окне → +1 чек к балансу |
| `REJECTED` | ни телефона, ни имени, либо дата не та → внятная причина в ответ |
| `MANUAL` | имя есть, телефона нет (OCR смазал цифры) → админу с кнопками ✅/❌ |

Чек на **старый** номер получает `REJECTED`, а не `MANUAL`:

```
❌ Вы оплатили на старый номер 996702760896. Деньги ушли не туда.
   Актуальный реквизит: Азиз Амираев, 996708760011
```

Подтверждать тут нечего — платёж ушёл другому человеку. Ручная проверка
остаётся только для случая «имя совпало, цифры не читаются».

**Телефон — основной критерий, не имя.** Имя в чеках пишется как
`Азиз А.` / `АЗИЗ А.` / `AZIZ A.` / `Азиз А. AZIZ A.`, а номер машинный и стабильный.
Блоки отправителя (`Плательщик`, `Отправитель`, `Реквизиты плательщика`) вырезаются
до всех проверок — иначе чек, где Азиз является **плательщиком**, ошибочно пройдёт.

Один и тот же чек нельзя засчитать дважды: `tx_id` под уникальным индексом.

Проверено на чеках MBank, DemirBank, Эльдик (RSK), Bakai и O!Деньги.

## Как работает каталог

`/lunch` создаёт опрос **из текста сообщения**. Каталог подключается после и не может
опрос сломать — `enrich()` обёрнут в `try/catch`.

Сопоставление строки меню с каталогом:

1. точное совпадение нормализованного ключа → берём блюдо, подтягиваем фото
2. `levenshtein ≤ 2` → спрашиваем админа: «„Бозо лагман" — это „Босо лагман"?»
3. иначе → блюдо дня, `dish_id = NULL`, живёт только внутри опроса

Нормализация снимает регистр, `ё/е`, эмодзи, `U+200E`, `+`, двойные пробелы.

**Порог 2, не 3.** На 13 реальных меню ближайшие *разные* блюда — `босо лагман`
и `гюро лагман` — находятся на расстоянии 3. Триграммный `similarity()`
здесь не работает вовсе: `шамси гарнир` / `курица с гарниром` = 0.62
при 0.91 у `бозо` / `босо`. Разделяющего порога не существует.

Фото хранятся как Telegram `file_id`, а не файлами. Отправка стоит одну строку.

## Структура

```
all/      опрос, парсер, Telegram-клиент, роутинг апдейтов
receipt/  извлечение текста, верификация чека, очередь ручной проверки
dish/     каталог, матчинг, алиасы, FSM админа
```

Порядок обработчиков в `TelegramUpdateWorker.handleMessage` важен:

- фото в **группе** от админа → каталог
- фото в **личке** → чек, кроме случая с явной подписью `/dish ...`
- диалог каталога не поглощает команды и сбрасывается через 10 минут,
  иначе забытое «Добавить блюдо» съело бы следующее сообщение админа в чате

## Известные ограничения

- `file_id` привязан к боту. Смените бота — фото придётся перезалить.
  `photo_unique_id` переживает смену и нужен для сверки «то же самое фото».
- `sendMediaGroup` не принимает `reply_markup`: фото и кнопки — разные сообщения.
  Альбом режется по 10 штук автоматически.
- `dish_question` не чистится. Несколько десятков строк в год, не проблема.
- Дубли фото по `photo_unique_id` не проверяются: колонка есть, сравнения нет.

## Смена реквизита

```
/recipient 996708760011 Азиз Амираев
```

В группе, от админа чата. Работает сразу, рестарт не нужен.

Прежний номер автоматически уезжает в `recipient.phone.legacy`. Чек на него
**отклоняется** с явным объяснением — не засчитывается и не уходит на ручную
проверку. Иначе бот подтверждал бы оплату, которой не было.

Регэксп имени строится из «Имя Фамилия» автоматически и ловит все написания,
встреченные в реальных чеках: `Азиз А.`, `АЗИЗ А.`, `AZIZ A.`, `Азиз А. AZIZ A.`,
`Азиз Амираев`, `Амираев Азиз`. Другой Азиз (`Азиз Байболов`) под шаблон не попадает.

## Развёртывание на сервере

### Память

| Компонент | В покое | Пик |
|---|---|---|
| JVM (`-Xmx256m`) | ~250 МБ | ~400 МБ |
| Postgres 16 | ~60 МБ | ~150 МБ |
| tesseract (только на время OCR) | — | +150–250 МБ |
| PDFBox рендер 300 dpi | — | +40 МБ |

Пик — когда кто-то прислал фото-чек: JVM и tesseract работают одновременно.

- **1 ГБ** — работает впритык. Нужен swap 1 ГБ, иначе OOM-killer прибьёт Postgres во время OCR.
- **2 ГБ** — комфортно, тюнить нечего. Рекомендую.

CPU: одного ядра достаточно, OCR одного скриншота — 1–3 секунды.
Диск: образ с tesseract ~450 МБ, данные за год — единицы мегабайт
(фото не хранятся, только 60-символьные `file_id`). 10 ГБ с запасом.

`docker-compose.yml` уже настроен под сервер: `mem_limit`, ротация логов,
порты наружу не проброшены.

### Перенос на другую машину

```bash
# на старой
docker compose exec postgres pg_dump -U postgres lunch_bot_db > dump.sql

# на новой
docker compose up -d postgres
docker compose exec -T postgres psql -U postgres lunch_bot_db < dump.sql
docker compose up -d
```

Не потеряйте `dish.photo_file_id` — без дампа все фото придётся переснимать.

**Остановите старый экземпляр перед запуском нового.** Два бота с одним токеном
воруют апдейты друг у друга: `getUpdates` отдаёт каждый апдейт только одному.

### Бэкап

```cron
0 3 * * 0 cd /opt/lunchbot && docker compose exec -T postgres \
    pg_dump -U postgres lunch_bot_db | gzip > /backup/lunch_$(date +\%F).sql.gz
```

## Оплата: баланс чеков

Один одобренный PDF-чек = право на одно блюдо. «Добавить ещё блюдо» требует
ещё одного чека. Считаются **чеки, не сумма** — у части людей сумма другая,
на неё бот не смотрит.

- проголосовал за блюдо → один чек «потрачен»
- отменил выбор → чеки возвращаются, можно выбрать другое блюдо без доплаты
- в конце дня опрос закрывается, неистраченные чеки **сгорают**: завтра нужен новый
- один и тот же чек дважды не пройдёт (`tx_id` уникален) — два блюда = два чека

Только **PDF**. Фото и скриншоты отклоняются с просьбой прислать PDF.
OCR убран полностью, поэтому приём чека — десятки миллисекунд, ноутбук не греется.

## Нагрузка

Бот использует **long polling**: одно висящее соединение вместо опроса каждую
секунду. На холостом ходу нагрузка близка к нулю, батарея ноутбука не садится.
Пиков от распознавания нет — чеки только PDF.
